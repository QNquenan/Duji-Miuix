# DuJi 项目工作流规则

## 规则 1：任务结束后检查并提交 git

每次任务完成后检查是否有文件变更：

- **有变更** → `git add -A && git commit -m "前缀: 中文摘要"`
- **无变更** → 跳过，不提交

commit message 必须使用统一前缀加中文摘要，例如 `fix:`、`feat:`、`refactor:`、`style:`、`docs:`。除非用户明确指定，否则不要省略前缀。

为什么：每个任务阶段的关键变更都要有清晰的 git 记录，但空提交没有意义；统一前缀也方便后续检索和整理历史。

## 规则 2：全程使用中文

所有推理过程和最终回答都使用简体中文。代码、标识符、文件路径、shell 命令、技术术语保留原文不翻译。


## 规则 3：使用Miuix组件库

所有可以使用Miuix组件库的都应该优先使用Miuix库，没有的先看能否用Miuix的组件库拼一个，实在不行就向我提问

## 规则 4：先提问澄清需求

请你在回答前先向我提问，要求一次只问一个问题；请根据我的回答继续追问，直到你有95%的信心、完全理解我的真实需求和目标时，再给出最终方案，否则就继续追问。

## 规则 5：每次都要提交项目

每次提交需要commit message 保持 `feat:`、`fix:` 等前缀格式
